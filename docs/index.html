The problems you know

If you've built a video data pipeline for robotics ML, you've hit these.

18 TB of extracted JPEGs

Your 1000-hour manipulation dataset is 2 TB as video. Pre-extract to frames? Now it's 18 TB, millions of tiny files, and a filesystem nightmare. frameforge reads directly from compressed MP4.

Training crosses episode boundaries

Your sampler picks frames 2995-3010. Frames 2995-2999 are "pick up cup", frames 3000-3010 are "open drawer". Your policy learns garbage. EpisodeSampler prevents this.

Silent DataLoader corruption

Open video in __init__, fork 8 workers, get silent data corruption. The #1 video ML debugging nightmare. frameforge uses lazy init + pickle-safe readers. Workers just work.

Rewrite pipeline per backend

PyAV, decord, torchcodec, OpenCV — different APIs, tensor formats, seeking behavior. With frameforge: change one argument. backend="pyav" → backend="torchcodec".

Random seeks decode 125 frames

GOP-250 video: seeking to one frame decodes 125 throwaway frames. frameforge's index cache + small-GOP encoding guide cut this to single digits.

Wrist + overhead + third-person

Multi-camera rigs need time-aligned frames across views. sync_readers_by_timestamp gives you synchronized tensors from all cameras in one call.

Code examples

From a single frame read to a full training DataLoader.

Basics

Read frames with Pythonic indexing

VideoReader auto-selects the best available backend. Access frames with integer indexing, slicing, or fancy indexing — just like a numpy array. All outputs are (H, W, C) uint8 tensors. 

from frameforge import VideoReader

reader = VideoReader("episode_001.mp4")

# Single frame
frame = reader[0]              # (H, W, C)

# Clip via slicing
clip = reader[100:116]        # (16, H, W, C)

# Fancy indexing
frames = reader[[0, 50, 100]]  # (3, H, W, C)

# Swap backend with one argument
reader = VideoReader("ep.mp4",
    backend="torchcodec", device="cuda:0")
Sampling

Temporal clip sampling

Four strategies: uniform (evaluation), random (augmentation), dense (action detection), and episode-aware (robotics). The episode sampler respects task boundaries so you never sample across different demonstrations. 

from frameforge.sampling import (
    UniformSampler,
    RandomSampler,
    EpisodeSampler,
)

# 16 evenly spaced frames
uniform = UniformSampler(num_frames=16)
clip = uniform.sample(reader)

# Random 2-sec window, 8 random frames
rand = RandomSampler(
    num_frames=8, clip_duration_sec=2.0)
clip = rand.sample(reader)

# Respect episode boundaries
ep = EpisodeSampler(
    episode_boundaries=[0, 3000, 6000],
    num_frames=16)
clip = ep.sample(reader)
Training

Worker-safe DataLoader

VideoClipDataset opens readers inside __getitem__, not __init__. No forked file handles, no silent corruption. Collate pads variable-length clips automatically. 

from torch.utils.data import DataLoader
from frameforge.dataloader import VideoClipDataset
from frameforge.dataloader.collate import video_collate
from frameforge.sampling import UniformSampler

dataset = VideoClipDataset(
    video_paths=episode_files,
    sampler=UniformSampler(16),
    backend="pyav",
    transform=lambda x: x.float() / 255.0,
)

loader = DataLoader(
    dataset, batch_size=8,
    num_workers=4, collate_fn=video_collate)

for batch in loader:
    # (B, T, H, W, C) feed to model
    loss = model(batch)
Multi-cam

Multi-camera synchronization

Read time-aligned frames from multiple camera feeds. Common in manipulation rigs with wrist, overhead, and third-person views. 

from frameforge.reader import VideoReader
from frameforge.sampling.sync import (
    sync_readers_by_timestamp,
)

readers = {
    "wrist": VideoReader("wrist.mp4"),
    "overhead": VideoReader("overhead.mp4"),
}

synced = sync_readers_by_timestamp(
    readers,
    timestamps_sec=[0.0, 0.5, 1.0, 1.5],
)
# synced["wrist"]    -> (4, H, W, C)
# synced["overhead"] -> (4, H, W, C)

Training data loading

The primary use case. Random access into compressed video without pre-extracting to JPEG. Episode-aware sampling, worker-safe DataLoaders, backend swapping to find the fastest decode path for your hardware.

Data curation

Scan metadata across 1000s of videos. Verify frame counts match annotations. Extract specific clips by index or timestamp. Re-encode with ML-friendly settings (-g 30 -bf 0). Multi-camera alignment checks.

Post-collection encoding

Re-encode raw recordings with optimal settings for ML — smaller GOP for fast seeking, constant framerate, faststart flag. Downscale 4K datasets to 720p. Not a recording stack replacement, but a re-encoding tool.

Offline evaluation

Read specific frames from rollout recordings. Compare model predictions frame-by-frame. Write prediction overlays back to video with VideoWriter for sharing and debugging.

Pipeline benchmarking

Before committing to a backend for a large training run, measure throughput, latency percentiles, and memory on your actual hardware and videos. Share HTML reports with the team.

Key takeaway: Low-level decoders (torchcodec, decord) handle bytes-to-pixels. Platforms (LeRobot, DALI) handle end-to-end workflows. frameforge is the missing middle layer — robotics-aware decode, sampling, and loading that works with any backend and any dataset format.
