1. Containers vs Codecs

A codec (coder-decoder) compresses video data. A container packages compressed streams into a file. The codec is the language; the container is the envelope.

2. H.264 — I/P/B Frames & GOP

H.264 compresses video as Groups of Pictures (GOPs). Each GOP starts with a keyframe and contains three frame types with very different sizes.

Frame types

  • I-frame — Self-contained, like a JPEG. The only frame you can decode independently. 5–10x larger than P-frames.
  • P-frame — Motion vectors + residual from previous frames. 30–50% the size of an I-frame.
  • B-frame — References past AND future frames. Smallest, but causes decode order ≠ display order.
The B-frame problem for ML: B-frames mean decode order ≠ display order. Seeking to "frame 5" can return the wrong frame. For datasets, encode with -bf 0 (zero B-frames). Compression penalty is only ~10–15%.

Macroblock encoding pipeline

H.264 divides frames into 16x16 macroblocks (8,160 per 1080p frame). Each goes through:

  1. Prediction — intra (spatial) or inter (temporal via motion vectors, quarter-pixel precision)
  2. Residual — subtract prediction from actual pixels
  3. Transform — integer DCT converts residual to frequency domain
  4. Quantization — the lossy step. Higher QP = more loss = smaller file
  5. Entropy coding — CABAC (10–15% more efficient) or CAVLC

3. Why Random Access is Slow

ML training samples random frames. Video codecs are built for sequential playback. To access any frame, the decoder must find the nearest keyframe before it and decode every frame in between.

4. H.265/HEVC Improvements

H.265 replaces H.264's 16x16 macroblock with a Coding Tree Unit (CTU) up to 64x64, recursively subdivided.

FeatureH.264H.265
Max block size16x1664x64
Blocks per 1080p8,160510
Intra prediction modes935
Max transform size8x832x32
In-loop filtersDeblockingDeblocking + SAO
Compression vs H.264Baseline30–50% better
Encode speed (software)30–60 fps5–15 fps
When to use H.265: Storage-constrained large datasets, 4K+, encode-once-read-many. Not when you need max decode compatibility.

5. Encoding: Bitrate, CRF & Quality

  • CBR — Fixed bits/sec. Wastes bits on easy frames. For streaming.
  • VBR — Allocates more to complex scenes. Better quality per byte.
  • CRF — Targets constant perceptual quality. Best for datasets.

6. Chroma Subsampling & Color

Human vision is more sensitive to brightness than color. Chroma subsampling stores color at lower resolution — but for robotics, this can silently corrupt training signal.

Interactive — Chroma comparison
Bright pixels have independent color data. Dimmed pixels share color with neighbors.
Manipulation tasks: In 4:2:0, a thin red wire on a green PCB loses its edge color — border pixels get chroma averaged between red and green. Use yuv444p if color accuracy matters.

Color space pitfalls

  • BGR vs RGB — OpenCV decodes BGR; PyTorch expects RGB. Swapped channels degrade accuracy silently.
  • BT.601 vs BT.709 — Different YUV→RGB matrices. Wrong matrix shifts reds warmer, greens off.
  • Limited vs Full range — Mismatched range (16–235 vs 0–255) crushes blacks and clips whites.

8. Why This Matters for Robotics

Egocentric video

Head-mounted cameras: high motion, variable lighting, rolling shutter. Use CRF 18–22, GOP 30–60.

Multi-camera rigs

4-camera 1080p30 at CRF 20 = ~40–80 Mbps total, 18–36 GB/hour. Hundreds of hours = multiple TB. H.265 saves 30–50%. Synchronize at constant framerate.

Dataset storage at scale

Frame-accurate access for training

With GOP=250, seeking to a random frame decodes 125 frames on average — a batch of 32 costs 8 seconds. Solutions:

  1. Small GOP (10–30) at encode time
  2. Frame index — pre-scan keyframe offsets, cache
  3. Sequential clips — sample contiguous clips, amortize cost
  4. Prefetch & cache — workers hide latency behind parallel I/O

frameforge implements strategies 1, 3, and 4 out of the box.

9. Common Pipeline Pitfalls

Variable frame rate (VFR)

Webcams, phones, GoPros produce VFR. Frame intervals vary. Assuming constant framerate silently misaligns frames with IMU/joint data.

Fix: Re-encode with -vsync cfr -r 30 before using in a pipeline.

Seek accuracy varies by backend

BackendAccuracyNotes
PyAVAccurateSeeks to keyframe, decodes forward. Most control.
decordAccurateBuilds frame index on open. Best for random access.
torchcodecAccuratePTS-based, handles reordering.
OpenCV±1–3 framesCAP_PROP_POS_FRAMES unreliable with B-frames.

Fork safety

Python multiprocessing uses fork() by default on Linux. Open a video in parent, children inherit corrupted state. Use spawn start method, or only open videos inside workers — which is what frameforge's VideoClipDataset does.

10. Recommended Encoding Settings

ffmpeg -i input.raw -c:v libx264 \
  -preset slow \
  -crf 18 \
  -profile:v high \
  -g 30 \
  -bf 0 \
  -pix_fmt yuv420p \
  -movflags +faststart \
  -vsync cfr \
  output.mp4
FlagWhy
-preset slowBetter compression. Encode once, read many.
-crf 18Near-lossless. Best for archival.
-g 30GOP = 1 second. Good seek performance.
-bf 0No B-frames. Display = decode order.
-movflags +faststartIndex at file start for fast seeking.
-vsync cfrForce constant framerate.
Color-critical tasks (wire routing, colored grasping): use -pix_fmt yuv444p -profile:v high444. Test hardware decode support first.
